## How to read this document

This is the top-level requirements document for OpenGate. It exists to anchor every subsequent technical document in the project (the Product Feature Document, the System Architecture, the System Design, the Database Schema, and the implementation plan), and to give any future reader — most importantly a hiring manager evaluating the project, secondarily the author six months later — a clear and honest understanding of what OpenGate is, what it demonstrates, and what it explicitly is not.

The document is written as a hybrid between a traditional Product Requirements Document and a Reference Implementation Specification. The structural sections (problem statement, goals, scenarios, success criteria) follow the convention of a standard PRD so the document remains familiar to anyone trained in product management or engineering documentation, but the content of each section is calibrated to the actual purpose of the project, which is a portfolio sample, not a commercial product. Readers familiar with classical PRDs will notice that several sections that would normally cover pricing, go-to-market, customer success, and competitive positioning are intentionally short or absent. This is not an oversight; those concerns are explicitly out of scope for this artifact and are addressed in a separate commercial project by the same author.

Subsequent documents in the OpenGate documentation set will progressively narrow from this high-level requirements view toward concrete implementation specifications. The Product Feature Document breaks the system into functional capability areas. The System Architecture document defines component boundaries and inter-component contracts. The System Design document covers the specific distributed-systems patterns the project demonstrates. The Database Schema document specifies the persistent data model including the event store, the read models, and the multi-tenant isolation strategy. The Implementation Plan finally decomposes the work into epics, user stories, and a fourteen-day initial task list. Each subsequent document treats the decisions made in this PRD as fixed inputs and only revisits them if a concrete implementation discovery invalidates an assumption made here.

## 1. Project framing and honest disclosure

OpenGate is a portfolio sample. It is a technical artifact whose primary purpose is to demonstrate the author's senior-engineer-level distributed systems competence in the Go programming language, in support of remote European senior Go engineering job applications during a sixty-day window beginning May 2026. It is not a commercial product, has no paying customers, has no plans to acquire customers within the project window, and integrates with no real hardware. The choice to be explicit about this framing in the document itself is deliberate. A reader who opens the GitHub repository and finds a document that pretends OpenGate is a product about to be launched will, correctly, be skeptical. A reader who finds a document that openly states the portfolio purpose, lists the architectural patterns demonstrated, and bounds the scope honestly will read the rest of the project with the appropriate frame of reference.

The author has a separate, much larger commercial project under development in parallel — a multi-tenant platform for event and service management — which carries the commercial product ambitions of the author. That project has a multi-year horizon and is not in scope for this document. OpenGate exists alongside that project specifically because building portfolio artifacts and building commercial products optimize for incompatible constraints. A portfolio artifact must be small enough to be reviewed in fifteen minutes by a busy reader, technically excellent in a narrow set of demonstrable patterns, and honest about its scope. A commercial product must address real customer pain, evolve over years in response to market feedback, and earn revenue. Trying to make one artifact serve both purposes typically results in something that is too unfinished to convince customers and too cluttered with non-essential commercial concerns to convince technical reviewers. OpenGate solves this by being unambiguously the former.

## 2. Target reader and success criteria

The primary intended reader of the OpenGate repository, taken as a whole, is a technical hiring manager, principal or staff engineer, or engineering director at a European or remote-friendly engineering organization who is evaluating the author for a senior Go backend or full-stack engineering role. The author is targeting organizations in the salary range of seventy thousand to one hundred and ten thousand euros gross per year for senior individual contributor roles, primarily in identity, access management, distributed systems, developer infrastructure, or platform engineering teams. Representative target employers include companies like Cast AI, Grafana Labs, Spacelift, Hygraph, Stedi, Monzo, lakeFS, and similar venture-backed product or infrastructure companies with remote European hiring practices. The document and the artifact it specifies are calibrated to what reviewers at those organizations look for when evaluating senior candidates.

Success for OpenGate is operationalized as follows. A representative reviewer at a target employer, given approximately fifteen minutes to evaluate the repository, the README, and the architecture document, arrives at the judgment that the author can independently design and implement a non-trivial distributed system in Go on senior architectural foundations, including event sourcing, CQRS, multi-tenant data isolation, idempotent command processing, distributed coordination, and observability. A reviewer should be able to identify within those fifteen minutes at least eight to ten specific senior-level architectural patterns deliberately implemented in the codebase, and should be able to point to corresponding tests that exercise those patterns under realistic conditions including failure modes. A reviewer should leave the evaluation with the impression that the author is capable of operating in their organization at the senior level the role advertises, and should have at least one specific topic from the project that would be productive to discuss in a technical interview.

Secondary readers include the broader Go community, for whom the repository functions as a worked example of the patterns it implements; the author's future self, who may extend, refactor, or harvest components from OpenGate for unrelated future projects; and any prospective collaborator who might join the project after the portfolio window closes. The document is written so that each of these secondary readers can also extract appropriate value, though all design decisions in conflict are resolved in favor of the primary reader.

What success is explicitly not measured by. OpenGate is not measured by acquired customers, monthly recurring revenue, free-tier signups, marketing reach, social media engagement, GitHub stars, or any other metric that would be appropriate for a commercial product. The single binary success criterion that matters is whether the artifact accomplishes the hiring signal it is built to send. Vanity metrics related to the repository — stars, forks, mentions — are welcomed when they happen but are not pursued and are not part of the success criteria.

## 3. Domain selection and rationale

OpenGate uses the domain of access control for small-to-medium fitness venues — independent gyms, CrossFit boxes, climbing gyms, and martial arts schools — as the substrate on which to demonstrate its architectural patterns. The choice of domain is justified along three axes, and the author considered alternative domains before settling on this one.

The first justification is authenticity. The author has approximately eight years of production engineering experience at a small Serbian company specializing in physical access control, ticketing, and integration with cameras and NFC readers. Real production systems shipped by the author include a stadium access control deployment for a Serbian first-league football club, an automatic license plate recognition parking system using Hikvision cameras, and event ticketing with Mifare NFC validation. The OpenGate domain narrative thus draws on direct, verifiable, hard-won expertise rather than inferred or recently-acquired knowledge. A reviewer reading the architecture document and the inline justifications for various design decisions will encounter reasoning grounded in physical-world constraints — for example, the way reader hardware fails in the field, the way credentials must remain valid for short offline windows when network connectivity drops, the way audit logs must be queryable in court-relevant time windows — that would be difficult to fake without the underlying experience.

The second justification is technical fit. The access control domain naturally and unavoidably exercises the exact set of distributed systems patterns the project intends to demonstrate. Access control is inherently event-driven, because each card tap, each door open, each credential issuance, each schedule change is a discrete event whose causal order matters for security and audit purposes. It is inherently multi-tenant, because a system that hosts multiple gyms must isolate their data and policies absolutely. It is inherently idempotency-sensitive, because the same physical card tap can produce duplicate network requests under flaky connectivity and the system must not double-process them. It is inherently hardware-abstraction-friendly, because real-world deployments encounter many different reader models and protocols and the application layer must not be coupled to any specific one. And it is inherently audit-log-critical, because security incidents, member disputes, and compliance audits all depend on the system being able to answer with confidence the question of who accessed what when. Each of these properties of the domain creates a natural justification for the corresponding architectural pattern. The patterns therefore appear in the codebase because the domain requires them, not because the author chose to demonstrate them in a contrived way.

The third justification is reviewer accessibility. The access control domain is concrete enough that any reviewer, regardless of whether they have prior experience with it, grasps the basic use cases in seconds. A member taps a card, a door opens, a log entry is written, an admin can see who entered when. There is no prerequisite domain knowledge required to evaluate the architecture, unlike domains such as financial derivatives, medical imaging, or supply chain optimization, which can require significant onboarding before a reviewer can judge whether the architecture is appropriate. This accessibility is valuable because the limited reviewer time budget should be spent evaluating engineering, not learning the problem space.

Domains that the author considered and rejected for the portfolio sample include the following. A generic distributed job queue or scheduler was rejected because the space is crowded with high-quality open source implementations (Temporal, River, Faktory) against which a sixty-day solo effort would compare unfavorably. An AI infrastructure or agent tooling project was rejected because the author does not have prior production AI infrastructure experience and would be visibly building catch-up work against well-funded incumbents. An event ticketing or parking domain was considered, both being adjacent to the author's expertise, but were rejected as portfolio projects because the author intends to use related domain expertise in the separate commercial multi-tenant platform and prefers not to dilute the unique signal of that future product. Gym access control sits at the intersection of authentic expertise, technical fit, and reviewer accessibility, with the additional benefit of being a domain the author is not planning to commercialize directly, which removes any temptation to dilute the portfolio artifact with commercial considerations.

## 4. Architectural patterns demonstrated

The deliberate purpose of OpenGate is to exercise a specific set of distributed systems and senior-engineering architectural patterns in a way that a technical reviewer can examine and verify. This section enumerates those patterns at the requirements level, with a one or two paragraph explanation of why each pattern is present and what domain need it addresses. The detailed implementation of each pattern is the subject of the System Design document; here we establish only the requirement.

Event sourcing is the foundational persistence pattern. All state changes in the system are recorded as immutable, append-only events in a Postgres-backed event store. Current system state is derived by replaying or projecting events. The justification from the domain is that access control systems have absolute audit log requirements. A regulator, a court, or a customer dispute may at any time require the system to answer with full fidelity what happened, in what order, on what authority. Event sourcing makes this property a structural guarantee rather than a feature to be implemented. The pattern also enables temporal queries, retroactive corrections via compensating events, and the ability to replay history into new read model shapes as the domain evolves.

Command-Query Responsibility Segregation, abbreviated CQRS, separates the model used for processing state-changing commands from the models used to answer queries. Commands flow through aggregates that enforce business invariants and emit events. Queries are served from read models that are materialized asynchronously from the event stream, optimized for the specific access patterns of the queries that read them. The justification is that access control systems have very different requirements for writes versus reads. Writes must be transactional, ordered, and consistent within an aggregate boundary. Reads, however, are dominated by audit queries that span months of history across millions of events, and by real-time dashboard queries that aggregate access patterns by member, by door, by time window. Trying to serve both kinds of access patterns from a single normalized model produces a system that is slow at queries and complex at writes. CQRS allows each side to be optimized independently.

Hexagonal architecture, often called ports and adapters, organizes the application so that the domain core is fully isolated from infrastructure and adapters. The domain defines abstract interfaces, called ports, for everything it needs from the outside world — the hardware reader, the controller, the persistence layer, the message bus, the clock. Adapters implement these ports for specific technologies. The justification is twofold. From the domain side, hexagonal architecture protects the domain logic from the volatility of integrations; reader hardware can be swapped, the persistence backend can be migrated, the clock can be replaced with a fake for tests, without the core access control logic changing. From the portfolio side, hexagonal architecture is the only honest way to demonstrate a hardware-coupled system without owning the hardware. OpenGate ships with two reader adapters in the repository — a fully deterministic fake used in unit tests, and a simulated adapter that mimics realistic field behavior including network latency, transient failures, and offline periods — but the interface it implements is the same interface a real DESFire EV3 or OSDP-protocol adapter would implement. A reviewer can verify the abstraction by reading the port definition and observing that no implementation detail leaks into the domain.

Multi-tenant data isolation via the shared-schema-with-tenant-identifier pattern, combined with Postgres Row-Level Security policies, ensures that data from one tenant cannot leak into queries for another tenant even in the presence of application bugs. Each row in each table carries a tenant identifier, and Row-Level Security policies are configured so that database connections operating on behalf of a specific tenant can only see rows belonging to that tenant. The justification is that a single security mistake in an access control system that exposes one gym's member data to another gym's administrator is a catastrophic failure mode. Defense in depth at the database level, beyond the application-level tenant filter, materially reduces the probability of such failures. The pattern is currently the recommended default in the industry for SaaS systems with fewer than approximately one hundred tenants, beyond which schema-per-tenant or database-per-tenant strategies may become preferable.

Idempotent command handling via client-supplied idempotency keys ensures that retried commands, whether retried by the client due to network failure or by an internal middleware due to timeout, produce exactly one observable effect rather than duplicate effects. The justification comes directly from the domain. A reader hardware that has lost network briefly will retry its authentication request when connectivity returns. A mobile app submitting a credential issuance command will retry on network failure. Without idempotent command handling, these retries produce duplicate audit log entries, duplicate access events, or duplicate credential records, each of which is a real bug with security and operational consequences.

Distributed coordination via Postgres advisory locks handles the small number of operations that genuinely require mutual exclusion across multiple instances of the application. Examples include exclusive credential generation when a credential identifier must be guaranteed unique across all in-flight requests, and the exclusive election of a single worker to perform scheduled maintenance operations. The justification for using Postgres advisory locks rather than introducing a separate coordination service such as Zookeeper or etcd is operational simplicity. The system already requires Postgres for its primary persistence; reusing it for the small number of coordination needs in the project avoids introducing an additional moving part. This choice is correct for a system of the scale OpenGate operates at and is documented as such so that a reviewer understands the trade-off.

Reliable webhook delivery with exponential backoff, configurable retry budget, and dead-letter queue handles the propagation of access control events to external systems. In the OpenGate domain, this represents the propagation of credential changes to door controllers, the notification of administrative systems about access events, and the integration of access logs into external compliance systems. Each webhook delivery is retried on failure according to an exponential backoff schedule with a finite retry budget; after the retry budget is exhausted the delivery is moved to a dead-letter queue for inspection. The justification is that webhook recipients fail unpredictably and the access control system must continue to operate correctly even when downstream consumers are temporarily unavailable.

Background job processing via the River queue library executes asynchronous work such as the materialization of read models from events, the delivery of webhooks, the periodic refresh of credential validity windows, and other operations that do not need to complete synchronously with the command that triggered them. River is chosen because it is a transactional, Postgres-backed job queue that fits the architectural decision to keep Postgres as the single persistence dependency, and because it has reached production stability with version one and active maintenance as of the project window.

End-to-end OpenTelemetry tracing instruments every request flowing through the system, from the inbound HTTP layer through the command handlers, through the event store writes, through the asynchronous read model materializations, through the webhook deliveries. A reviewer using the included Grafana Tempo configuration can visualize a complete request lifecycle and verify that the system has been instrumented at the level of attention that production systems require.

Embedded role-based access control via the Casbin library handles authorization within the application layer. Casbin is chosen over alternatives such as OpenFGA or SpiceDB because it is the appropriate weight for the scale of OpenGate, embedding directly into the Go process without requiring a separate service. The justification mirrors the Postgres-advisory-lock decision; operational simplicity for a small system is itself a senior engineering judgment that the project demonstrates.

Contract testing with versioned API schemas, expressed in OpenAPI 3.1, ensures that the contract between the backend and any client — including the included React admin dashboard and any hypothetical future mobile or third-party client — is explicit, validated, and version-managed. The OpenAPI schema is the source of truth from which both the server-side request validators and the client-side type definitions are generated. The justification is that production systems need contract stability across deployments, and the practice of generating both ends from a single schema definition is the senior-engineer standard for achieving this.

Graceful shutdown and zero-downtime deployment semantics ensure that when a running OpenGate process receives a termination signal, it stops accepting new requests, completes in-flight requests, drains the background job worker pool, flushes its observability buffers, and exits cleanly within a configurable timeout. The justification is that any system intended to be deployed and operated will eventually be restarted, redeployed, or scaled, and a system that drops in-flight requests or corrupts state on shutdown is not production-quality regardless of any other architectural sophistication it possesses.

## 5. Use case scenarios

The system must support the following end-to-end scenarios. Each scenario describes a coherent user-perceptible interaction that exercises several of the architectural patterns listed in the previous section. The scenarios are written here in narrative form to make them accessible to non-engineering readers; their decomposition into formal user stories with acceptance criteria is the subject of the Implementation Plan document.

The first scenario covers member enrollment. An administrator, authenticated into the admin dashboard, opens the member management view for their gym and creates a new member record. They enter the member's name, contact information, and membership type. They issue an access credential to the member by associating a card identifier with the member record. They configure access permissions specifying which doors the member may use and during which time windows. The system processes the administrator's command as a sequence of events written to the event store, materializes the events into the read models that back the admin dashboard, and emits a webhook to any subscribed external systems notifying them of the new member and credential. The administrator sees the new member appear in the dashboard within a small number of milliseconds, even though the read model materialization is asynchronous.

The second scenario covers a member attempting access at a door. The simulated reader at a door sends an authentication request to the OpenGate backend including the card identifier and the door identifier. The backend looks up the member associated with the card, evaluates the access permissions for that member at that door at the current time, and returns either a grant decision or a deny decision. An access event is written to the event store recording the attempt, its outcome, and its reasoning. The decision must be returned within a strict latency budget — for the portfolio sample, a target of fifty milliseconds at the ninety-ninth percentile is sufficient — because real-world door hardware has limited patience before timing out.

The third scenario covers offline reconciliation. A simulated door controller, having lost network connectivity for a configurable duration, comes back online and synchronizes with the backend. During the offline period it has continued to make local access decisions based on a cached credential set, and has recorded local access events. The reconciliation protocol exchanges the local events for the period of disconnection, the backend deduplicates them against any duplicates it may have already received, and updates the audit log. Credential changes that occurred at the backend during the offline period are propagated to the controller. The scenario must complete without data loss, without duplicate events in the audit log, and without inconsistent access decisions remaining in the system.

The fourth scenario covers immediate revocation. An administrator revokes a member's access, perhaps because of a payment failure or a discipline issue. The revocation event is written to the event store. The system propagates the revocation to all affected door controllers via webhook with appropriate retries. Crucially, any in-flight authentication attempt at any door that the backend evaluates after the revocation timestamp must result in a deny decision, even if the door controller's local cache has not yet received the revocation update. This is a subtle requirement that the system's architecture must support; the test suite must include scenarios that verify the property.

The fifth scenario covers audit queries. A compliance officer, authenticated into the admin dashboard, opens the audit log view and queries for all access events at a specific door during a specific thirty-day window. The query must return results within milliseconds even when the underlying event store contains millions of historical events, because the read model is optimized for exactly this query pattern. The compliance officer can drill into a specific event to see its full context including the reasoning for the grant or deny decision. The audit results are verifiable against the underlying event log by an engineer who reproduces the read model materialization logic.

The sixth and final scenario covers tenant data export. An administrator, exercising their tenant's right to data portability, requests a complete export of all data associated with their gym. The system produces a structured export including the event log, the materialized read models, and the configuration. The export is signed with a verifiable signature so that its integrity can be confirmed by a third party. The export format is documented and re-importable into a fresh OpenGate instance, supporting the case where a customer wishes to migrate their data to a self-hosted instance.

## 6. Non-goals

A document specifying a portfolio sample is most useful when it states explicitly what is excluded as well as what is included. Without an explicit non-goals section, the temptation to expand scope is constant and the project drifts toward something that cannot be finished in the available time. The following items are explicitly out of scope for the sixty-day window, and any of them appearing in the actual repository would be a deviation from this specification rather than an enhancement of it.

Real hardware integration is out of scope. The system implements two adapters for reader hardware, both of which are software simulations, one deterministic and one realistic. No code path in the project will be tested against a physical DESFire reader, OSDP controller, Hikvision camera, or any other piece of hardware during the project window. A future maintainer may add such adapters by implementing the documented reader port interface, but the burden of that work is explicitly deferred.

A mobile application for members is out of scope. The system exposes API endpoints that a mobile app could use, and those endpoints are designed with mobile usage in mind, but no React Native, Flutter, or native mobile implementation is included in the project. The included frontend is a web-based admin dashboard only.

Payment processing, billing, subscription management, and any commercial transaction handling are out of scope. The system has no concept of a paid plan, no Stripe or PayPal integration, no invoicing. Members exist as administrative records; how their gym charges them is none of OpenGate's concern in this version.

GDPR compliance certification is out of scope. The patterns chosen for the system — Mifare card-based access, no biometric processing, multi-tenant data isolation, full audit logging, support for data export — are aligned with the principles of GDPR compliance and would form an appropriate technical foundation for a commercial product that pursued certification. However, the project does not include the legal documentation, data processing agreements, Data Protection Impact Assessments, or other artifacts that an actual certified compliant system would require. A reader who is interested in the GDPR-aligned design choices will find them documented in the architecture document, but should not assume the system is itself a compliant product.

Production deployment infrastructure beyond Docker Compose is out of scope. The project ships with a docker-compose configuration sufficient to bring up the system locally on a developer machine or a single Linux server. Kubernetes manifests, Helm charts, Terraform modules, multi-region deployment topologies, blue-green deployment automation, and similar production operations artifacts are not included.

Customer-facing commercial materials are out of scope. There is no marketing website, no pricing page, no privacy policy, no terms of service, no customer documentation portal. The README of the repository is the only customer-facing document, and it is calibrated for an engineering audience rather than a buyer audience.

Real customer data, real members, real gyms, and real anything are out of scope. All data used in demonstrations, screenshots, and tests is synthetic. Names, addresses, identifiers, and any other personally identifiable fields in the sample data are generated. No real human is enrolled in any instance of the system.

## 7. Scope and timeline

The project window is sixty days, beginning at the time of the author's departure from her current employer, with an expected work capacity of forty to sixty hours per week. The total available engineering capacity is therefore approximately four hundred and eighty to seven hundred and twenty hours, of which a budget of approximately five hundred and fifty hours is assumed as the realistic working figure after accounting for unavoidable interruptions.

The work is structured into four milestones of approximately fourteen days each, plus a four-day finishing pass at the end. Milestone one, days one through fourteen, focuses on the core domain implementation: the event store, the aggregates, the command and query handlers for the central workflows, the in-memory test infrastructure, and the contract test suite scaffold. Milestone two, days fifteen through twenty-eight, focuses on the persistence layer including the Postgres-backed event store, the materialization of read models, the multi-tenant isolation including Row-Level Security policies, and the background job processing via River. Milestone three, days twenty-nine through forty-two, focuses on the integration layer including the webhook delivery service, the simulated reader adapter, the OpenTelemetry instrumentation, and the API layer with OpenAPI specification. Milestone four, days forty-three through fifty-six, focuses on the React-based admin dashboard, the integration with the backend, the architecture documentation, and the polished README. The final four days are reserved for end-to-end testing, demonstration video recording, polishing of documentation, and submission of the repository to relevant communities.

The detailed task-level decomposition of each milestone is the subject of the Implementation Plan document and is not specified here. What is specified here is that the milestones are end-to-end rather than horizontally sliced. Each milestone produces a system that, although less feature-complete than the milestone after it, is internally coherent and demonstrable. This sequencing decision is deliberate; it ensures that even if the project window is cut short for any reason, the artifact at any milestone boundary is a complete-feeling slice rather than a half-built layer.

## 8. Technology stack

The technology choices for OpenGate are made deliberately to balance senior-engineer credibility, learning surface during the project window, and production realism. The choices and their justifications follow.

For the backend language and runtime, Go 1.26 is used, pinned via the `go` directive in `go.mod`, which is the single source of truth and is consumed by CI through `setup-go`'s `go-version-file`. Go is the explicit target language of the portfolio signal and is the foundation of the entire project. The realized 1.26 satisfies the original target — most recent stable release as of project start, no lower than 1.23 — and ensures access to recent standard library improvements.

For the relational database, PostgreSQL version 16 is used. The choice of Postgres is foundational to several of the architectural decisions described in section four, including the event store, the Row-Level Security multi-tenant isolation, and the use of advisory locks for coordination. Version 16 is chosen for stability, broad ecosystem support, and access to recent features such as logical replication improvements.

For database access from Go, the sqlc library is used to generate type-safe Go code from raw SQL queries. This is paired with the pgx driver for low-level Postgres connectivity. The combination is chosen over alternatives such as GORM or Ent because it produces the most performant and most reviewable code, with no reflection-based runtime overhead and with all queries visible as plain SQL in the repository. A reviewer evaluating the repository can read the SQL directly without translating from an ORM abstraction.

For background job processing, the River library is used. River is a Postgres-backed transactional job queue specifically designed for Go, reaching version one stability in the project window. The choice keeps the system's runtime dependencies limited to Postgres alone in the persistence and coordination layer.

For HTTP routing, the Chi library is used. Chi is a lightweight, idiomatic Go router that integrates cleanly with the standard library net/http package and supports middleware composition in a clear way.

For authorization, the Casbin library is used embedded directly in the Go process. Casbin is the appropriate weight for a system of OpenGate's scale and supports role-based and attribute-based access control via a model file plus a policy store.

For observability, OpenTelemetry is the tracing and metrics instrumentation library, with traces exported to Tempo via the OpenTelemetry Collector and metrics scraped by Prometheus. Grafana is used as the visualization layer. This stack is chosen because it is the current production standard across the Go ecosystem.

For the frontend, Next.js using React 19 is used. The choice of Next.js over alternatives such as Vite plus React Router is made because Next.js is the dominant production React framework in 2026 and a reviewer is most likely to be familiar with it. The frontend uses Tailwind CSS for styling, in line with the author's preference for utility-first styling on professional artifacts. Component primitives are provided by the shadcn/ui library, which provides accessible, customizable, copy-paste React components built on Radix UI primitives.

For API contracts, OpenAPI 3.1 is the schema language. Code generation tools produce Go server-side validators and TypeScript client-side types from the schema.

Languages and tools that were considered and rejected for the project include the following, with reasons. GORM was rejected for runtime reflection overhead and for producing less reviewable code than sqlc. Ent was considered but rejected because its learning surface within the sixty-day window competes with other learning that produces stronger signal. MongoDB was considered as a read model store but rejected because the project already requires Postgres and adding a second persistence technology dilutes the focus. Redis was considered as a cache layer but rejected as unnecessary for the read patterns of a portfolio sample. NATS or RabbitMQ were considered as message buses but rejected in favor of using Postgres notification channels and River for the asynchronous work the system requires.

## 9. Hardware abstraction strategy

Because OpenGate has no real hardware, but the access control domain is inherently hardware-coupled, the project must take a position on how it represents hardware. The position taken is that the hardware boundary is defined by clean Go interfaces in the domain layer, with multiple adapter implementations behind those interfaces, none of which require physical devices.

The two adapters included in the repository are the fake adapter and the simulated adapter. The fake adapter is fully deterministic and is used in unit tests and in scenarios where reproducibility is required. It responds to method calls with pre-programmed responses, supports test-controlled time advancement, and produces no random behavior. The simulated adapter is calibrated to mimic the behavior of realistic field hardware. It introduces small but bounded latencies, simulates occasional transient failures with a configurable failure rate, simulates network disconnection and reconnection on a controllable schedule, and produces realistic-looking event streams. The simulated adapter is the one used in integration tests, in the demo video, and in any reviewer's local exploration of the system.

The reader port interface, defined in the domain layer, captures what the application needs from any reader implementation. A simplified sketch of the interface includes operations for authentication of a presented credential, for healthcheck queries, for the receipt of credential updates pushed from the backend, and for the streaming of access events back to the backend. Any real reader adapter — whether for DESFire EV3 over OSDP, for a Wiegand legacy reader through a gateway, for an ESP32-based DIY reader, or for a Hikvision camera with NFC capability — would implement this same interface.

The architecture documentation discusses this approach explicitly and includes a guide for the future implementation of a real adapter. The guide is included because the question of how OpenGate would integrate with real hardware is the most likely concrete technical question a reviewer asks during an interview about the project, and the documentation should answer it preemptively.

## 10. Constraints and assumptions

The development environment is assumed to be Linux, specifically Ubuntu, matching the author's primary development machine. The system is developed and tested on this platform first. Compatibility with macOS for development is a goal but not a guarantee for the project window; compatibility with Windows is not pursued.

The deployment target is a single-host Linux server using Docker Compose. The system is not designed within this window to be deployed to Kubernetes, to serverless platforms, or to multi-host configurations, although the architectural choices made do not preclude such deployments in the future.

The runtime dependencies are limited to PostgreSQL and the OpenGate application binary itself, plus the optional observability stack consisting of the OpenTelemetry Collector, Tempo, Prometheus, and Grafana. No paid SaaS services are required to run the system. The observability stack is optional in the sense that the system runs without it, although the recommended demonstration mode includes it.

The system is intended to be self-hostable from day one. There is no SaaS-only mode; every component runs on the user's own infrastructure if they choose to run it there.

The open source license is Apache 2.0, chosen because it is the most permissive of the licenses that retain patent protections and is the most familiar license to enterprise reviewers. Other licenses considered included MIT, which was rejected for lack of patent protection, and AGPL, which was rejected as inappropriately restrictive for a portfolio sample.

The repository is hosted on GitHub. A backup mirror on a self-hosted Gitea instance is considered as a stretch goal but is not in the critical path of the project.

## 11. Risks

A portfolio project has its own risks distinct from a commercial product, and they deserve enumeration so they can be monitored throughout the project window.

The first risk is scope creep into commercial product features. The temptation to add billing, to design landing pages, to begin marketing the project, or to court real customers will be present throughout the project window. The mitigation is the explicit non-goals section of this document. Any contemplated addition that does not appear in the goals must be evaluated against the question of whether it materially improves the hiring signal, and most do not.

The second risk is over-engineering relative to portfolio purpose. The patterns implemented are valuable and worth implementing well, but the project must not become an exercise in implementing every distributed-systems pattern that exists. Several patterns that would normally appear in a production system at scale, such as eventual-consistency conflict resolution or cross-region replication, are deliberately excluded from this project because they would not be exercised meaningfully at the scale OpenGate operates at and would distract from the patterns that are exercised.

The third risk is abandonment if the job hunt resolves successfully before the project window closes. If the author lands an offer at week six of the eight-week project window, the question arises whether to continue building OpenGate or to direct the remaining attention elsewhere. The mitigation, decided in advance, is that the project should be brought to a clean stopping point at the next milestone boundary in such a case, with the README updated to honestly describe the status of the project, rather than continuing to build under reduced attention or abandoning it incomplete.

The fourth risk is demo collapse if the simulated hardware adapter does not feel realistic enough to a reviewer. The mitigation is to invest enough time in the simulated adapter to make its behavior compelling, including realistic latencies, realistic failure modes, and realistic event streams. The simulated adapter is part of the artifact and must be reviewed by the author with the same standards as the rest of the code.

The fifth risk is dependency drift if any of the chosen libraries publish breaking changes during the project window. The mitigation is to pin all dependencies to specific versions at the start of the project and to avoid major version upgrades during the window unless specifically required.

The sixth risk is frontend complexity exceeding backend complexity. The portfolio signal is primarily about senior Go backend competence; if the React dashboard becomes the bulk of the project, the signal is diluted. The mitigation is to keep the dashboard scope tightly bounded to what is necessary to demonstrate the backend's capabilities, and to use the shadcn/ui component library to minimize custom frontend code.

## 12. Open questions and decisions deferred

A number of decisions are intentionally not made in this document because they belong properly to subsequent documents in the documentation set. They are listed here so they are not forgotten.

Whether each specific read model is materialized synchronously within the command transaction or asynchronously after the transaction commits is a decision deferred to the System Design document. Different read models have different freshness requirements and the choice may be made per read model.

The exact partitioning strategy for the event store as the system scales, including whether to partition by tenant or by time window or both, is deferred to the Database Schema document.

The specific policy for handling concurrent credential issuance, including whether to use optimistic concurrency control via event versioning or pessimistic locking via advisory locks, is deferred to the System Design document.

The exact format and signature scheme for the tenant data export, including which cryptographic primitives are used, is deferred to the System Design document.

The detailed shape of the admin dashboard's information architecture, including the page layouts and the navigation structure, is deferred to the Product Feature Document.

The choice of OpenTelemetry semantic conventions for the access control domain, including the naming and attribute structure of the spans that the system emits, is deferred to the System Design document.

The specific approach for the demonstration video that will accompany the project, including its length, its structure, and its narration approach, is deferred to a separate marketing-adjacent document closer to the project's end.
