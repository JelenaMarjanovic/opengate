## How to read this document

This document is the second layer in the OpenGate documentation set. It sits between the Product Requirements Document, which fixed the goals, scope, non-goals, architectural patterns to be demonstrated, and use case scenarios at the requirements level, and the System Architecture document, which will define concrete component boundaries, inter-component contracts, and deployment topology. The Product Feature Document occupies the intermediate space in which the system is decomposed into functional capability areas, each of which is described at a level of detail sufficient for a reader to understand what the system does at a feature level without yet committing to how each feature is implemented at the architectural or code level.

The decomposition principle used here is by bounded context in the domain-driven design sense, adapted to the relatively small scope of a portfolio sample. Each capability area corresponds to a coherent slice of the access control domain that owns its own data, exposes its own operations, and serves an identifiable subset of the system's use case scenarios. The boundaries between capability areas are drawn to minimize cross-area coupling in the steady state, while accepting that any non-trivial system has some unavoidable cross-area interactions, which are themselves first-class citizens of this document.

Two parameters fixed at the start of this document govern its scope and style, and the reader should be aware of them before reading further. The first parameter is the narrower-scope discipline: every feature described in this document must trace back to at least one of the twelve architectural patterns enumerated in section four of the PRD or to one of the six use case scenarios enumerated in section five of the PRD. Features that would belong in a hypothetical commercial product but do not exercise any of the architectural patterns are excluded from this document. The author explicitly accepted that several features which would be obvious additions to a real product, such as member self-service portals, payment status tracking, photo-on-file storage, or branded report exports, are excluded on this basis. The second parameter is the hybrid dashboard treatment: rather than describing the admin dashboard as a single isolated capability area or distributing its description so thinly across backend modules that it loses coherence, each backend capability area in this document carries a short Dashboard touchpoints sub-section describing the user interface surfaces that interact with that capability, and a single cross-cutting Admin Dashboard section near the end describes the elements of the dashboard that do not belong to any individual capability area, principally information architecture, navigation, authentication flow, and rendering policies.

The audience for this document is the same audience as for the PRD: primarily a technical hiring manager or principal engineer at a target employer evaluating the OpenGate repository as portfolio material, and secondarily the author's future self and any future collaborator. Readers familiar with PRDs and feature specifications from companies practicing structured product management — Stripe, Atlassian, Shopify, and similar organizations are referenced here as the documentation style being emulated — will find the format familiar. Anti-fluff discipline is applied throughout; marketing adjectives, repetitive introductory filler, and sentences that do not convey technical value are omitted.

## 1. System decomposition overview

OpenGate is decomposed into ten primary functional capability areas and two cross-cutting infrastructure capabilities. The primary capability areas are tenant provisioning and identity, member management, credential issuance and lifecycle, access policy and permissions, reader and door operations, access authorization decisions, offline reconciliation, audit log and compliance queries, webhook delivery, and tenant data export. The two cross-cutting capabilities are observability and operations, and background job processing. A single dashboard cross-cutting section follows the capability areas and covers concerns that do not belong to any individual one.

The relationship between these capability areas is best understood by tracing a single member's journey through the system. A tenant is provisioned, an administrator authenticates into the tenant, the administrator creates a member, the administrator issues a credential to the member, the administrator assigns an access policy to the member, the member presents the credential at a reader on a door, the system makes an authorization decision and records it, the audit log absorbs the decision into its read model, and any external system subscribed to the tenant's webhooks receives notification of the access event. If the door's local controller has been offline during any portion of this sequence, the reconciliation protocol bridges the temporary disconnection. If at any later point the tenant's owner exercises the data export right, every event in the foregoing sequence appears in the export package.

Each capability area in the following sections is described according to a consistent structure. The structure begins with a brief functional description of what the capability does and what value it contributes to the overall system, continues with the dashboard touchpoints that surface the capability to the administrator user, identifies the architectural patterns from PRD section four that the capability exercises and the use case scenarios from PRD section five that it serves, and concludes with an explicit list of features that a reader might expect to find under the capability but that are out of scope for this version. The patterns-and-scenarios mapping is consolidated at the end of this document into a traceability matrix.

## 2. Capability area: Tenant provisioning and identity

The tenant provisioning and identity capability is the foundational substrate on which every other capability operates. In the OpenGate domain, a tenant represents a single gym, climbing center, CrossFit box, or martial arts school. Administrative users belong to a tenant, members are scoped within a tenant, doors are configured within a tenant, audit logs are partitioned by tenant, and every other entity in the system carries the tenant identifier of its owning gym. The capability owns the lifecycle of tenants from creation through configuration, the lifecycle of administrative users within each tenant, the authentication of those users when they log in, and the enforcement of role-based authorization on every operation that the application layer exposes.

Tenant creation is handled out-of-band by a command-line bootstrap tool rather than by a self-service signup flow, because the portfolio sample does not need to demonstrate a self-service onboarding experience and because the cost of building one would be disproportionate to its contribution to the hiring signal. The bootstrap tool, invoked by an operator with database access, creates a new tenant record, assigns it a stable identifier, configures its initial settings, and creates the first administrative user for the tenant with the owner role. From that point onward, the owner can log into the dashboard and invite additional administrators by issuing them credentials with assigned roles.

The administrative role model is intentionally small. Three roles are defined: owner, manager, and auditor. The owner role grants every operation within the tenant, including configuration changes, the addition of further administrators, and the initiation of tenant data exports. The manager role grants member and credential management, access policy configuration, and viewing of audit logs, but not configuration of tenant settings or invitation of additional administrators. The auditor role grants only read access to members, credentials, policies, doors, and audit logs, intended for compliance reviewers who must inspect the system without making changes. Authorization is enforced by Casbin policies loaded at application startup from a Casbin model file and a policy table in the database. Every request handler in the application checks the requesting user's role against the requested operation before proceeding.

Authentication uses session cookies established after a username-and-password login. Passwords are stored as Argon2id hashes with per-password salt and tuned cost parameters. Sessions are opaque tokens stored in the database with associated user identifier, tenant identifier, role, and expiry timestamp. Logout invalidates the session immediately. Session timeout is configurable per tenant with a default of sixty minutes of inactivity. Password reset is performed by another administrator with appropriate role; there is no email-based self-service password reset in this version because building a reliable email delivery pipeline is itself a non-trivial engineering exercise that would not strengthen the demonstrated patterns.

The multi-tenant data isolation property is enforced through two layers operating in defense-in-depth. The application layer always includes the requesting session's tenant identifier as a filter on every database query, mediated by repository code generated by sqlc from SQL queries that include tenant identifier in their where clauses. Beneath this, Postgres Row-Level Security policies are configured on every table that carries a tenant identifier column, such that database connections operating on behalf of a tenant cannot return rows belonging to any other tenant regardless of what the application layer requests. The connection's tenant identifier is set as a Postgres session variable on connection checkout from the pool. If application code somewhere were to omit the tenant filter from a query through a bug, the Row-Level Security layer would still return no rows or raise an error, depending on the exact policy mode. This defense-in-depth approach is the central feature of the capability area from the perspective of the portfolio signal.

**Dashboard touchpoints.** The capability surfaces in the dashboard through the login page, which establishes the session and tenant context; the session indicator in the persistent navigation bar showing the current administrator's name, current tenant name, and current role; the user management page where the owner can invite, modify, or remove administrative users; and the tenant settings page where the owner can configure session timeout, webhook signing secret rotation policy, and the tenant's display name and contact information. The login page handles credential errors with explicit messaging that distinguishes between unknown user, wrong password, and locked-out account.

**Patterns exercised.** This capability exercises multi-tenant data isolation via the shared-schema-with-tenant-identifier pattern combined with Postgres Row-Level Security policies, and embedded role-based access control via Casbin. The graceful shutdown pattern also touches this capability because session-invalidating shutdowns must propagate to all clients with active sessions.

**Scenarios served.** This capability is exercised by every scenario in PRD section five, because every scenario begins with an authenticated administrative session and every scenario produces data that must remain isolated to its tenant.

**Explicitly out of scope.** Self-service tenant signup, single-sign-on via SAML or OIDC, multi-factor authentication, federated identity, password reset via email, member-facing authentication, and multi-tenant administrator switching where a single human identity has accounts across multiple tenants are all excluded. The omission of multi-factor authentication in particular is worth noting: a real commercial access control product would treat MFA as table stakes, and OpenGate would also include it if it were a real product, but in the portfolio context the additional complexity does not exercise any of the twelve architectural patterns being demonstrated and is therefore deferred.

## 3. Capability area: Member management

The member management capability owns the lifecycle of member records within a tenant. In domain terms, a member is a person who uses the gym and whose access is controlled by the system. Members do not authenticate to OpenGate themselves and are not users of the system; they are administrative records about people. Each member has a name, contact email, optional contact phone, a status field representing the member's current standing with the gym, a creation timestamp, and an opaque external reference identifier provided at creation time that allows an external system, for example the gym's accounting software or member relationship management tool, to correlate the OpenGate record with its own representation of the same person.

The member status field is constrained to three values: active, suspended, and expired. An active member has full access according to whatever policies are assigned. A suspended member is temporarily blocked from access without losing their associated credentials or policy assignments, typically because of a payment issue or disciplinary action that the gym staff expects to resolve. An expired member has had their membership lapse and is treated as denied for access purposes; expired status is also a useful retention state for compliance because audit log queries about past access can still resolve the member's identity. Status transitions are commands processed by the member aggregate, which validates that the requested transition is legal (an active member can become suspended or expired, an expired member cannot become suspended, and so on) and emits a status-changed event.

Members are created through an administrator-issued command on the dashboard. The command flow is the canonical example of the system's write path: the request arrives at the API layer, is validated for shape and authorization, is passed to the member command handler, is processed by the member aggregate which enforces invariants such as uniqueness of email within the tenant and validity of the external reference format, and produces one or more events that are written to the event store within a single transaction. After the transaction commits, the events are picked up by the projector workers which update the member read models that back the dashboard's list and detail views.

The read models for member management are optimized for the specific queries the dashboard makes. The member list read model is keyed by tenant and includes only the fields needed for list display: identifier, name, email, status, and the timestamp of the last access event by the member. The member detail read model is keyed by tenant and member identifier and includes the full member record plus the count and most recent timestamp of associated credentials and the count of access events within the last thirty days. Search by name or email uses a separate search read model indexed by trigrams or a similar substring-friendly structure, the exact choice of which is deferred to the System Design document because it depends on Postgres extension availability constraints not yet resolved at the PFD level.

**Dashboard touchpoints.** The capability surfaces in the dashboard through the member list page with search, filter, and pagination controls; the member detail page showing the member's identity fields, status history, associated credentials, associated access policies, and recent access events; the member creation form; and the member edit form for changing identity fields and status. The member detail page is the most frequently visited page in the dashboard in steady-state operation, so it must remain responsive under realistic data volumes, with skeleton-based loading states for the slower asynchronous sub-views like associated credentials and recent access events.

**Patterns exercised.** This capability is the cleanest demonstration of event sourcing and CQRS in the system. Every change to a member is an event in the event store; every view of a member in the dashboard is served from a read model materialized asynchronously from those events. The capability also exercises multi-tenant data isolation because member records are tenant-scoped and the Row-Level Security policies on the members table enforce that boundary.

**Scenarios served.** This capability is the primary substrate of use case scenario one, member enrollment, and is exercised by scenarios two and four because access decisions and revocation events reference the affected member.

**Explicitly out of scope.** A member self-service portal where the member can update their own contact details, a payment status integration where the system tracks whether the member's dues are paid, a photo-on-file storage system where each member's face can be associated with their record, a family-account hierarchy where multiple member records can be linked under a single billing relationship, a freezing schedule mechanism where a member's status can be set to suspended automatically for a planned vacation period, and a member tagging system for arbitrary administrator-defined categorizations are all excluded. Several of these would be required for a commercial product but none of them exercises any pattern that is not already covered by the simpler member CRUD lifecycle.

## 4. Capability area: Credential issuance and lifecycle

The credential capability owns the association between physical access tokens and members. In the simulated OpenGate world, a credential is represented as a UUID standing in for what would in a real deployment be a Mifare Classic 1K serial number, a Mifare DESFire EV3 application identifier plus card serial, or a similar physical card identifier. Each credential is associated with exactly one member at exactly one tenant. A member may have multiple credentials, for example a primary keycard plus a backup, but a credential identifier in the system is unique per tenant. The capability owns the issuance of new credentials, the revocation of existing credentials, the modification of credential expiry dates, and the query of credential status at any past or current point in time.

Credential issuance is the canonical example of idempotent command handling in the system. An administrator submitting an issuance command provides an idempotency key, typically generated client-side as a UUID and persisted in the dashboard until the command is acknowledged. If the dashboard's network request fails and is retried, the same idempotency key is supplied, and the backend recognizes that the command has already been processed and returns the original result rather than issuing a second credential. The idempotency key is stored alongside the credential record with a long retention period, sufficient to outlive any reasonable client retry budget. This idempotency property is mandatory and not optional: physical card readers in real deployments are notorious for triggering retries under flaky connectivity, and a system that allowed double issuance would produce duplicate credential identifiers, which would in turn produce ambiguous audit logs and undetected security holes.

The credential lifecycle consists of four states: issued, active, revoked, and expired. A newly created credential is issued; it transitions to active immediately if no future activation date is configured, or remains issued until the configured activation date is reached. An active credential can be used at readers. Revocation is immediate and event-sourced; once a credential is revoked, it cannot be reactivated, and any subsequent access decision for that credential returns deny regardless of policy. Expiration occurs automatically when the configured expiry date is reached, and is also immediate from the perspective of access decisions. The four-state lifecycle is intentionally minimal and does not include a paused or held state, because such states would add complexity without exercising any additional pattern.

Concurrent credential issuance creates the requirement for distributed coordination because the system must guarantee that the credential identifier is globally unique within the tenant. In the most common case, where the administrator scans an already-known card identifier into the issuance form, the uniqueness check is a simple database constraint. However, the system also supports a mode where the backend generates a credential identifier on behalf of the administrator, for use cases where the gym is enrolling members in bulk and the administrator does not want to scan each card individually but instead wants to print a batch of identifier-stickered cards from the backend's generated list. In this generation mode, multiple concurrent issuance commands could theoretically conflict on the next-available identifier within the tenant's credential space. The system handles this conflict by taking a Postgres advisory lock keyed on the tenant identifier plus a known constant during the generation operation, ensuring that only one issuance command can be generating a new identifier per tenant at a time. The lock is held only for the duration of identifier allocation, not for the entire credential issuance command, so its contention impact is bounded.

**Dashboard touchpoints.** The capability surfaces in the dashboard primarily on the member detail page, where the credentials sub-section shows the member's existing credentials and offers a credential issuance flow consisting of either entering a known identifier or invoking the backend generation mode. A separate credentials index page lists all credentials in the tenant with filter by status and search by identifier, intended for administrators investigating which member holds a specific physical card. The credential detail page shows the full lifecycle history of the credential including its issuance event, any modification events, and its revocation or expiration if applicable. Revocation is performed by an explicit revoke button on either the member detail page or the credential detail page, with a confirmation dialog because the operation is irreversible.

**Patterns exercised.** This capability is the primary demonstration of idempotent command handling and distributed coordination via Postgres advisory locks. It also exercises event sourcing because every credential lifecycle transition is an event, and CQRS because the dashboard's credential views are served from read models. The capability exercises multi-tenant isolation by ensuring that credential identifiers are unique per tenant rather than globally unique, which is the correct domain choice because two different gyms might independently issue cards with the same physical serial number.

**Scenarios served.** This capability is exercised by use case scenarios one (member enrollment includes credential issuance) and four (immediate revocation operates on credentials).

**Explicitly out of scope.** Programming of physical NFC cards through any kind of contactless writer interface, integration with a specific card brand such as HID or Nedap, lookup of card type from a presented identifier, dual-credential issuance flows where two cards are issued atomically for example to a member and their guardian, temporary visitor credentials with built-in expiry shorter than seven days, and a credential template system where common issuance parameters can be saved and reused are all excluded.

## 5. Capability area: Access policy and permissions

The access policy capability owns the rules that determine which members can access which doors during which time windows. A policy in OpenGate is a first-class entity associating one or more doors with one or more time windows, expressed in a structured form that the access decision engine can evaluate efficiently at runtime. Policies are assigned to members through a many-to-many relationship, allowing a member to hold multiple policies simultaneously and a single policy to apply to many members. The policy capability owns the creation, modification, and assignment of policies, and the evaluation of policies at access decision time is handled by the access authorization capability described in the next section.

The policy data model intentionally restricts the expressiveness of access rules to a level appropriate for a portfolio sample. A policy consists of a name, a tenant scope (always equal to the tenant that owns the policy, enforced by the schema), a list of door identifiers covered by the policy, and a list of time windows during which the policy grants access. A time window is a structured representation of weekdays plus a start time and end time in the tenant's local time zone. The same policy can include multiple time windows, for example one for weekday mornings and another for weekend afternoons. Holidays, calendar exceptions, and date-range bounds are not supported in this version because they would require integration with a holiday calendar source or a date-range expression evaluator, neither of which exercises any additional pattern beyond what the basic time-window model already exercises.

Policy evaluation at access decision time follows a deterministic algorithm. The decision engine collects all policies assigned to the member presenting the credential, expands their door coverage and time windows, and grants access if and only if at least one policy assigned to the member covers the door being accessed and contains a time window that includes the current tenant-local time. The default behavior in the absence of any covering policy is deny; this is the default-deny model that any non-trivial access control system requires. The algorithm is intentionally simple and deterministic to allow the decision rationale to be captured precisely in the access event, supporting the audit query capability's requirement that an auditor can understand exactly why a given decision was made.

A subtlety in the policy capability that bears attention is the moment-of-decision semantics. An access decision must use the policy state that exists at the moment the decision is made, not the policy state at some earlier moment such as the last time the credential synced to the controller. This is most consequential during the immediate revocation scenario from PRD section five: when a member's policy is removed, any subsequent access attempt evaluated by the backend must use the new state, even if the controller's local cache still reflects the old state. The system therefore distinguishes between backend-evaluated decisions, which always reflect current policy state, and controller-local decisions, which reflect the policy state at the time of the controller's last sync. The audit log records both kinds of decisions with explicit labeling of which evaluation occurred where.

**Dashboard touchpoints.** The capability surfaces in the dashboard through the policy management page where policies are listed, created, and edited; the policy editor where the door coverage and time windows of a specific policy are configured; the member detail page where policies assigned to the member are displayed and managed; and the effective-permissions preview on the member detail page where the administrator can see at a glance what doors the member can access right now according to the current policy assignments. The effective-permissions preview is a synthesized view computed on demand from the current policy state and serves as a debugging aid for administrators who are trying to understand why a member can or cannot access a particular door.

**Patterns exercised.** This capability exercises event sourcing because every policy creation, modification, and assignment is an event. It exercises CQRS because the dashboard's policy views are served from read models. It exercises aggregate invariant enforcement, a specific aspect of event sourcing, because policy modifications must validate constraints such as the requirement that every door identifier in the policy belongs to the tenant. It indirectly contributes to the multi-tenant data isolation property because policies are tenant-scoped.

**Scenarios served.** This capability is exercised by use case scenarios one (member enrollment includes policy assignment), two (access decisions evaluate policies), and four (revocation typically involves policy removal in addition to or instead of credential revocation).

**Explicitly out of scope.** A rule expression language or domain-specific language for complex access rules, holiday calendars and date-range exceptions, anti-passback rules where a credential cannot be used twice on the same door without an intervening exit, occupancy limits that reject access when a door's downstream area is at capacity, escalation policies that grant access only with co-presence of a second member, and a policy template system for common configurations such as "weekday business hours" are all excluded. The deliberately restricted policy model is itself the design decision being demonstrated.

## 6. Capability area: Reader and door operations

The reader and door operations capability owns the system's relationship with the physical hardware boundary, even though no physical hardware is involved in the portfolio sample. A door in OpenGate is a logical entity representing a controlled entry point, configured by an administrator with a name, a location description, and a current operational status. Each door has exactly one logical reader, which is the component that presents credentials to the system and on whose authentication request the access decision is made. The reader is represented in the application's domain layer as an abstract Go interface, the Reader port, which defines the operations the system needs from any reader regardless of its underlying technology.

The Reader port interface is the single most important design artifact in the hardware abstraction strategy and is described in detail here because its precise shape determines what kinds of reader implementations can be plugged in later without modifying any domain code. The interface includes operations for the authentication of a presented credential at a door, returning a decision plus a reason code; the receipt of credential and policy updates pushed from the backend, used to keep the reader's local cache fresh during periods of normal connectivity; the streaming of access events back to the backend, used both during online operation and during the reconciliation protocol after periods of disconnection; and the reporting of heartbeat signals carrying the reader's operational status. The interface explicitly does not include any operation tied to a specific physical protocol such as Wiegand, OSDP, or any vendor-specific NFC card type, because such operations would couple the domain to a specific technology and prevent later replacement.

Two reader adapter implementations ship with the OpenGate repository. The fake reader adapter is fully deterministic and is used in unit tests and in scenarios where reproducibility is paramount. It responds to authentication requests with pre-programmed responses, accepts test-controlled time advancement, produces no random or non-deterministic behavior, and is the default adapter used in the contract test suite that verifies the application layer against the Reader port. The simulated reader adapter, in contrast, is calibrated to mimic the behavior of realistic field hardware. It introduces small but bounded network latencies in the range of one to twenty milliseconds, simulates occasional transient failures with a configurable failure rate around half a percent, simulates network disconnection and reconnection on a controllable schedule that can be driven from the demo runner, and produces an event stream that resembles what a real Hikvision or DESFire deployment would emit under similar conditions. The simulated adapter is the one used in integration tests, in the demo video, and in any reviewer's local exploration of the system.

Door status is itself an event-sourced concept. A door can be in one of three states: online, degraded, or offline. The online state is the steady state when the reader is reachable and is reporting heartbeats within the expected interval. The degraded state is entered when the reader's heartbeat is delayed but still recent enough to be considered alive, typically because the simulated network is experiencing high latency. The offline state is entered when no heartbeat has been received within the offline threshold, configurable per door with a default of sixty seconds. Status transitions emit events that feed into the door status read model and trigger webhook deliveries to subscribed external systems.

**Dashboard touchpoints.** The capability surfaces in the dashboard through the door management page where doors are listed with their current operational status and last heartbeat timestamp; the door detail page showing the door's identity, its configuration, its recent operational history, and the access events that have occurred at the door in the recent past; the door creation and edit forms; and the manual override controls on the door detail page, which allow an owner-role user to issue an emergency unlock command. The manual override emits a distinct event type from a normal access decision because an emergency unlock is not a credential-driven decision but an explicit administrative action, and the audit log must distinguish the two unambiguously.

**Patterns exercised.** This capability is the primary demonstration of the hexagonal architecture, also known as the ports and adapters pattern. The Reader port interface defined in the domain layer is the port; the fake and simulated adapters are two implementations of the port. The architecture documentation includes an explicit guide for adding a third adapter for a real physical reader, because the question of how OpenGate would connect to real hardware is the most likely concrete technical question a reviewer asks during an interview about the project, and the architecture must answer it preemptively. The capability also exercises event sourcing through the door status events.

**Scenarios served.** This capability is the primary substrate of use case scenarios two (access attempts at a door) and three (offline reconciliation).

**Explicitly out of scope.** Real reader adapters for DESFire EV3, OSDP, Wiegand, Hikvision, ZK Teco, or any other physical hardware; door zoning where multiple doors are grouped into logical zones with collective access rules; anti-tamper detection where the reader reports physical interference with the hardware; environmental sensors such as door-open detection beyond the basic open/closed signal; and integration with external door controllers from companies such as HID or Honeywell are all excluded. The simulated adapter is the upper bound of hardware realism in this version.

## 7. Capability area: Access authorization decisions

The access authorization capability is the runtime heart of the system. Every other capability either provides the data this capability needs (members, credentials, policies, doors), records the output this capability produces (audit log), or coordinates the surrounding environment (reader operations, reconciliation). The decision flow is conceptually simple but has strict performance requirements and several subtle correctness properties that the implementation must respect.

The decision algorithm proceeds as follows. The reader at a door presents an authentication request including the door identifier and the credential identifier being authenticated. The application layer resolves the credential to a member, checks the member's status, checks the credential's status, evaluates the member's assigned policies against the door and the current time, and produces a decision. The decision is one of grant or deny, accompanied by a reason code drawn from a closed enumeration: granted, deny_unknown_credential, deny_revoked_credential, deny_expired_credential, deny_member_suspended, deny_member_expired, deny_outside_policy_window, deny_no_policy_covers_door, deny_door_offline_locally, and deny_emergency_lockout. The reason code is the audit-relevant explanation for the decision and must be returned in the response so that the reader can display an appropriate message and the audit log can record the full reasoning.

The performance target for the decision is a p99 latency of fifty milliseconds for the synchronous portion, measured from the moment the application layer receives the request to the moment the response is returned. This target is achievable because the decision can be served entirely from the materialized read models for member, credential, and policy state, without requiring a replay of events. The read models for these aggregates are maintained synchronously within the command transactions that change them, ensuring that a decision evaluated immediately after a write reflects the new state. The same read models also back the dashboard's effective-permissions preview, which means the dashboard preview is guaranteed to show the same effective state the decision engine would use.

Idempotency at the decision layer is a critical property because flaky network conditions between reader and backend cause retries. The decision endpoint accepts an idempotency key in each request and returns the same response on retry, even if the original response was lost. The idempotency cache stores the request hash and the response for a configurable window, defaulting to ten minutes, sufficient to cover any reasonable retry budget. The cache itself is a Postgres-backed table rather than an in-memory structure, because in-memory caches would not survive application restarts and would not be visible to other application instances if the system were horizontally scaled. Beyond the idempotency property, each decision is also event-sourced: the decision event is written to the event store synchronously within the decision-processing transaction, ensuring that the decision is durably recorded before the response is returned. This synchronous event write is the strongest correctness guarantee in the system, because it eliminates any window during which a decision could be returned but lost.

The decision event captures not only the decision and reason code but also the full context needed to reconstruct the reasoning later. The context includes the credential identifier presented, the door identifier targeted, the time of the decision in both UTC and tenant-local time, the member identifier resolved from the credential or null if unknown, the policy identifiers evaluated, the time window expressions that matched or failed to match, and the application version that performed the evaluation. The application version is included because policy evaluation logic might evolve over the lifetime of the system, and an auditor reviewing an old decision must be able to determine which version of the logic produced it.

**Dashboard touchpoints.** The capability surfaces in the dashboard through the real-time access event feed on the home page, which shows a streaming view of the most recent access decisions across the tenant, with grant decisions in neutral typography and deny decisions visually distinguished by the critical color from the brand palette; the access event detail page showing the full decision context including the reasoning chain; and the per-door and per-member access event sub-views on the door detail and member detail pages.

**Patterns exercised.** This capability exercises event sourcing through the synchronous write of decision events, CQRS through the read models that serve the decision, idempotent command handling through the decision idempotency cache, and end-to-end OpenTelemetry tracing because every decision is wrapped in a span that propagates from the reader through the application to the event store and out to any downstream subscribers.

**Scenarios served.** This capability is the primary substrate of use case scenario two (member attempting access at a door) and is exercised by scenarios three (reconciliation involves decisions made offline) and four (revocation must take effect immediately at the decision layer).

**Explicitly out of scope.** Face recognition or other biometric decision inputs, multi-factor decisions requiring credential plus second factor, occupancy-aware decisions where access depends on downstream space occupancy, two-person-rule decisions requiring co-presence of two members, and a machine-learning anomaly detection layer that scores decisions against historical patterns are all excluded. The deterministic algorithmic decision is itself the designed property.

## 8. Capability area: Offline reconciliation

The offline reconciliation capability handles the inevitable case where a controller loses network connectivity with the backend for a period of time and must continue to make access decisions locally based on its cached state. During the offline period, the controller relies on its last-synchronized snapshot of credentials, policies, and door configuration to make decisions, and records each access event locally with a stable local identifier and a timestamp. When connectivity returns, the controller and the backend execute a reconciliation protocol that transmits the locally accumulated events to the backend, deduplicates them against any events the backend may already have, and pushes any state changes that occurred at the backend during the offline window down to the controller.

The reconciliation protocol is designed around three correctness properties. The first property is no event loss: every access decision made locally during the offline period must appear in the audit log after reconciliation, regardless of any retries, partial failures, or interleaving with other reconciliations. The second property is no duplicate events: the same access decision must not appear twice in the audit log even if the controller, due to its own retry logic, transmits the same event multiple times. The third property is convergence: after reconciliation completes, the controller's local state must reflect every state change that occurred at the backend during the offline period, including credential issuances, revocations, policy modifications, and policy assignments, such that subsequent decisions at the controller are correct relative to the current backend state.

The deduplication mechanism exploits the idempotent command handling pattern. Each local access event carries a stable composite identifier consisting of the controller identifier and a controller-local event sequence number. The backend's reconciliation endpoint treats each transmitted event as an idempotent command keyed on the composite identifier. If the backend has already accepted an event with the same composite identifier, the second submission is acknowledged but produces no additional state change. This mechanism handles the case where the controller's transmission succeeds but the acknowledgement is lost in transit, causing the controller to retry transmission of the same event.

The push direction of reconciliation, where backend state changes are propagated to the controller, exploits the same event stream that drives the read model projectors. The controller maintains a watermark recording the last event from the backend it has applied locally, and on reconnection it requests all events in the tenant since the watermark. The backend serves these events from the event store with a tenant filter, the controller applies them to its local cache, and the controller advances its watermark. If the reconnection itself is unreliable and the controller disconnects again partway through reception, the watermark is only advanced for events fully applied, so the next reconnection resumes correctly.

A subtlety worth describing concerns the conflict between local decisions made during the offline window and backend state changes that occurred during the same window. Consider the case where a member's credential is revoked at the backend at time T-revoke, while the controller is offline and unaware of the revocation, and the member then presents the credential at the controller at time T-attempt where T-attempt is after T-revoke. The controller, using its stale cache, grants access locally. After reconciliation, the audit log shows the local grant decision and also shows the revocation event, with the revocation predating the access attempt. This is not a bug; it is the inherent reality of distributed systems with network partitions, and the audit log must surface the conflict clearly so that human review can address it. The dashboard's audit query view highlights such conflicts visually so an auditor reviewing access events at an offline-recovered door can immediately see the cases where local decisions diverged from what backend state would have produced.

**Dashboard touchpoints.** The capability surfaces in the dashboard through the controller health view, which lists all controllers across the tenant with their current connectivity state, last-sync timestamp, and count of pending unsynced events; the per-controller detail view showing the connectivity history of the controller; and the audit query view's conflict highlighting feature described in the previous paragraph.

**Patterns exercised.** This capability is the most distributed-systems-flavored capability in the system. It exercises idempotent command handling through the dedup mechanism, event sourcing through the use of the event store as the source for the push-down stream, and at the architectural level demonstrates a small but realistic eventually-consistent replication scheme.

**Scenarios served.** This capability is the primary substrate of use case scenario three (offline reconciliation).

**Explicitly out of scope.** A multi-controller-quorum protocol where multiple controllers vote on a decision in degraded scenarios, a peer-to-peer reconciliation mode where controllers can synchronize with each other rather than only with the backend, and an offline-first credential issuance mode where new credentials can be issued at the controller without backend round-trip are all excluded. The reconciliation protocol is single-controller-to-single-backend in this version.

## 9. Capability area: Audit log and compliance queries

The audit log and compliance queries capability owns the query side of the system's access event history. Where the access authorization capability produces decision events and writes them to the event store, the audit log capability is concerned with making those events efficiently queryable for compliance review, dispute resolution, and operational debugging. The capability is the cleanest demonstration of CQRS in the system: the write model for decisions is the event store; the read model for queries is a separately maintained denormalized projection optimized for the specific query patterns that compliance work requires.

The compliance query patterns supported in this version are filter by tenant (always implicit), filter by one or more doors, filter by one or more members, filter by decision outcome (grant or deny, or specific deny reason codes), filter by time window (start and end timestamps), and combinations of the above. The result is returned in reverse chronological order by default with pagination, and each result row includes the timestamp, the member's name and identifier, the door's name and identifier, the credential identifier presented, the decision outcome, the reason code, and a flag indicating whether the decision was evaluated locally at a controller or by the backend. The query also supports drill-down to the full decision event from any row in the result.

The read model is materialized asynchronously from the access decision event stream by a projector that runs as a River background job. The projector consumes events in order, indexed by tenant and time, and writes denormalized rows to the read model table. The denormalization includes joining the member's name and the door's name into the row, so that queries do not need to perform joins at read time. When member or door names change, a separate projector updates the affected rows in the read model. The freshness target for the read model is one second behind the event store under normal load; in degraded conditions the lag may grow but is reported as a metric so operators can detect and respond.

Audit query latency is the second performance-critical path in the system after the decision latency. The target for query latency is a p95 of two hundred milliseconds for a query returning a single page of results, even when the underlying event store contains millions of historical events. This target is achievable because the read model is indexed on the query dimensions (tenant + time, tenant + member + time, tenant + door + time) and because pagination uses keyset-based pagination rather than offset-based pagination, ensuring that query cost is independent of how deep into the result set the user has paged.

A subtle requirement of the audit capability is verifiability against the underlying event log. An engineer or auditor with access to the system must be able to take a sample of audit query results and verify that the read model matches what the event log would produce if replayed. The system supports this by exposing an internal-only endpoint that, given a specific time window, replays the events from the event store and produces the audit log rows that the projector would have produced, allowing comparison against the actual read model. The endpoint is not exposed in the dashboard; it is intended for forensic and verification use only.

**Dashboard touchpoints.** The capability surfaces in the dashboard through the audit query page with the filter and pagination controls described above; the audit event detail page showing the full decision context including the reasoning chain; and the export-to-CSV control on the audit query page for compliance reporting workflows. The CSV export is performed asynchronously through a River background job because large exports may take longer than a request can reasonably wait, with the resulting file available for download from a downloads page once the export job completes.

**Patterns exercised.** This capability is the primary demonstration of CQRS, with a clearly separate read model optimized for query patterns that the write side cannot satisfy efficiently. It also exercises event sourcing because the source of truth is the event store, and background job processing because the projector and export both run as River jobs.

**Scenarios served.** This capability is the primary substrate of use case scenario five (audit queries) and is also exercised by scenario three (reconciliation events appear in the audit log) and scenario six (data export reads from the audit log among other sources).

**Explicitly out of scope.** A natural-language query interface for compliance questions, alerting on suspicious access patterns, predictive analytics for member behavior, and a scheduled report subscription system where compliance reports are emailed weekly are all excluded. The audit query capability is a query-on-demand interface; automated scheduled querying is deferred.

## 10. Capability area: Webhook delivery

The webhook delivery capability handles the outbound propagation of OpenGate events to external systems subscribed by the tenant. In the domain, webhooks are the primary integration mechanism by which an OpenGate-using gym connects to its other software systems, such as accounting tools that should learn when a member becomes expired, customer relationship management tools that should learn when a member is created, or external monitoring systems that should learn when a door goes offline. The capability owns the subscription model, the delivery model, the retry model, and the dead-letter handling.

Each tenant can configure zero or more webhook subscriptions, each consisting of a target URL, a shared secret used for HMAC signing of payloads, and a filter specifying which event types the subscription receives. The event type filter is a set of named event types drawn from a closed enumeration including member_created, member_status_changed, credential_issued, credential_revoked, policy_assigned, access_event_recorded, door_status_changed, and tenant_export_completed. Wildcard filters are supported (all-events, or all-member-events) to accommodate subscribers that want to receive a broad subset without listing every type. Subscriptions are managed by administrators with the manager or owner role.

The delivery model is single-attempt-per-event with retry, not at-least-once exact delivery. When an event matching a subscription's filter is produced, a delivery job is enqueued in River with parameters identifying the subscription, the event, and the attempt number. The worker that processes the job constructs the payload, computes the HMAC signature, makes an HTTP POST request to the subscription's URL with a configurable timeout (defaulting to ten seconds), and considers the delivery successful if the response status is in the 2xx range. Any other outcome, including timeout, connection error, or 4xx or 5xx response, is treated as failure.

The retry model is exponential backoff with jitter. The first retry after a failure is scheduled approximately fifteen seconds later. Subsequent retries double the delay (thirty seconds, sixty seconds, two minutes, four minutes, and so on) up to a maximum of one hour, then continue retrying at one-hour intervals until the retry budget is exhausted. The retry budget defaults to twenty-four hours after the initial attempt. The jitter is randomized within plus or minus twenty percent of the scheduled delay to avoid thundering-herd retry storms when multiple subscribers fail simultaneously.

When the retry budget is exhausted, the delivery is moved to a dead-letter queue, implemented as a table containing one row per dead-lettered delivery with the full payload, the subscription identifier, the event identifier, and the history of failed attempts. Administrators can inspect the dead-letter queue from the dashboard and either retry individual deliveries manually or accept and discard them. Dead-lettered deliveries are retained for a configurable period (default thirty days) before being automatically purged.

The payload format and HMAC signature are deliberately designed to be familiar to anyone who has implemented webhook handling for Stripe or similar services, because following an established convention is itself a senior engineering signal. The payload is JSON with a versioned schema, a timestamp, the event type, the tenant identifier, and an event-specific data block. The HMAC signature is computed over the request body using SHA-256 with the subscription's secret and is transmitted in a request header. Subscribers verify the signature on their end before processing.

**Dashboard touchpoints.** The capability surfaces in the dashboard through the webhook subscriptions page where subscriptions are listed and managed; the subscription detail page where the URL, secret, and event filter are configured; the delivery status sub-view on the subscription detail page showing recent delivery attempts and their outcomes; and the dead-letter queue inspection view where dead-lettered deliveries can be inspected, retried manually, or discarded. Secret rotation is a first-class operation that generates a new secret while temporarily accepting payloads signed with either the old or new secret, allowing the subscriber to update their verification key without dropped messages.

**Patterns exercised.** This capability is the primary demonstration of reliable webhook delivery with exponential backoff, finite retry budget, and dead-letter queue, and a major use site for background job processing via River.

**Scenarios served.** This capability is exercised by use case scenarios one (subscribers may want to receive member creation events) and four (revocation propagation to external systems happens via webhook). All other scenarios that produce events also flow through this capability if subscribers exist.

**Explicitly out of scope.** A subscription marketplace where third parties can publish reusable subscription configurations, an inbound webhook capability where OpenGate receives events from external systems, a subscription transformation language that allows the subscriber to filter and reshape payloads in flight, and a real-time subscription protocol such as Server-Sent Events or WebSockets to external systems are all excluded. The single delivery model is HTTP POST.

## 11. Capability area: Tenant data export

The tenant data export capability supports the right-to-data-portability principle that any modern multi-tenant system must respect. An owner-role user can request a complete export of all data associated with their tenant, including the entire event log, the materialized read models, the configuration, and the metadata sufficient to reconstruct the tenant's state at any point in its history. The export is delivered as a structured archive in a documented format, and is signed with a verifiable signature so that the receiving party can confirm the integrity of the export.

The export is initiated by an explicit user action from the dashboard. The user clicks the export button, optionally specifies a time range to limit the export to events within that range, and confirms the request. The system enqueues an export job in River, queues the user back to the export status page, and runs the export asynchronously in the background. The export job iterates over the event store with a tenant filter, writing events in order into a structured archive, accompanied by the materialized read models in their current state and the configuration at the time of the export.

The archive format is documented in the project's architecture documentation and is designed to be re-importable into a fresh OpenGate instance. The format consists of a top-level manifest describing the export, the tenant identifier and metadata, the export timestamp, the version of OpenGate that produced the export, and a checksum of the contents. Beneath the manifest are subdirectories for events (organized by aggregate type and grouped by month), read models (one file per read model snapshot), and configuration (one file per configurable area). All files are JSON, gzipped for size, and the entire archive is tar-formatted.

The signature scheme uses Ed25519 detached signatures. Each OpenGate deployment maintains a long-term Ed25519 signing key in its configuration, distinct from any tenant-specific secrets, and signs the manifest of each export with this key. The corresponding public key is published at a well-known URL on the deployment, allowing any third party to retrieve the public key and verify the signature on the manifest. The integrity check covers the manifest, which itself includes checksums of all archive contents, so verifying the manifest signature implicitly verifies the entire archive.

A subtlety in the data export capability is the handling of in-flight events at the moment the export is taken. The system uses a consistent snapshot semantics, taking the event store position at the moment the export begins and exporting all events up to but not including any events committed after that position. Events that occur during the export run do not appear in the export but are available in any subsequent export. This approach avoids the alternative complexity of trying to capture a moving target while accepting that any export represents the state at the moment it was initiated, which is the semantics most external regulatory frameworks understand and accept.

**Dashboard touchpoints.** The capability surfaces in the dashboard through the data export page, accessible only to owner-role users, where pending and completed exports are listed; the export request form where the optional time range is specified; the export status indicator showing job progress; and the secure download link for completed exports, which is time-limited and tied to the requesting administrator's session.

**Patterns exercised.** This capability exercises event sourcing through the event store replay, CQRS through the inclusion of read model snapshots, background job processing through the asynchronous export job, and the signing scheme demonstrates an integration of cryptographic verification into an application-layer feature.

**Scenarios served.** This capability is the primary substrate of use case scenario six (tenant data export).

**Explicitly out of scope.** Selective export of only certain entity types, partial export with field-level redaction, scheduled recurring exports, multi-format exports (only the OpenGate native format is supported), and re-import functionality (the format is documented as re-importable but no re-import tool ships with the project) are all excluded.

## 12. Cross-cutting capability: Observability and operations

Observability is treated as a first-class capability in OpenGate even though it does not correspond to any single user-visible feature. The reasoning is that any system intended to be operated must answer with confidence the questions of what is the system doing right now, how is its performance trending, and where in the request path is a specific problem occurring. The observability capability provides the instrumentation that lets operators answer these questions, and demonstrates the OpenTelemetry-based approach that is the current standard across the Go ecosystem as of the project window.

The tracing layer instruments every request flowing through the system. An inbound HTTP request is wrapped in a server span, which propagates context through the command or query handler that processes the request, through the database operations the handler performs, through any background work enqueued in the same transaction, and into any webhook deliveries triggered by the resulting events. The trace context is propagated to external systems through standard W3C TraceContext headers in webhook deliveries, so that an integrated environment where the subscriber also runs OpenTelemetry can produce a complete end-to-end trace covering both systems.

The metrics layer instruments the operational properties that operators most often need to observe. The command-side metrics include command latency histograms keyed by command type, command error rate, and event store write rate. The query-side metrics include query latency histograms keyed by query type and read model freshness lag. The background job metrics include job execution latency, retry counts, and the depth of the pending queue. The decision-path metrics include access decision latency at the p50, p95, and p99 percentiles, decision rate by outcome, and reader heartbeat freshness. Metrics are exposed in Prometheus format on a dedicated endpoint and are scraped by the Prometheus instance bundled in the docker-compose stack.

The health check layer provides two endpoints. The liveness endpoint returns success if the application process is running and able to respond to HTTP requests, and is used by container orchestrators to detect a wedged process that should be restarted. The readiness endpoint returns success only if the application is fully initialized, can connect to the database, can connect to the River queue, and is not in the middle of a graceful shutdown sequence. The distinction between liveness and readiness is important because the orchestrator must not restart a process that is gracefully shutting down (liveness should still be true) but should also stop routing traffic to it (readiness should be false).

Graceful shutdown is the final element of the operations capability. When the application receives a termination signal, it stops accepting new HTTP requests by setting readiness to false; it allows in-flight HTTP requests to complete up to a configurable timeout (default thirty seconds); it stops the River job worker pool and waits for any in-flight jobs to complete or to be released back to the queue; it flushes the OpenTelemetry trace buffers; it closes the database connection pool; and only then does the process exit. If the configurable timeout is exceeded, the process exits anyway, but this is treated as an unclean shutdown and is logged at warning level.

**Dashboard touchpoints.** The observability capability has no direct dashboard touchpoints in the OpenGate admin UI itself. Operational observability is accessed through the Grafana instance bundled in the docker-compose stack, which connects to the Prometheus metrics source and the Tempo trace source. The system administrator opens Grafana to view dashboards that include the production-grade panels for command latency, decision latency, projector lag, webhook delivery success rate, and so on. The Grafana dashboard JSON ships in the repository as part of the observability configuration.

**Patterns exercised.** This capability exercises end-to-end OpenTelemetry tracing and the graceful shutdown pattern. It does not contribute its own architectural pattern beyond those, but is included as a first-class capability because the senior-engineering signal of the project includes the demonstration that observability is treated with the seriousness it deserves rather than being an afterthought.

**Scenarios served.** This capability is invisibly exercised by every scenario in PRD section five, because every action in the system produces telemetry that an operator can observe.

**Explicitly out of scope.** An in-application observability dashboard built into OpenGate itself, alerting rules and on-call paging integration, structured log aggregation beyond what the OpenTelemetry collector handles, and integration with commercial APM products are all excluded. The bundled Grafana, Prometheus, Tempo stack is the upper bound of observability tooling in this version.

## 13. Cross-cutting capability: Background job processing

Background job processing is the infrastructure under which several other capabilities operate. The capability owns the queue and worker model, the job scheduling, the failure handling, and the singleton coordination for jobs that must not run on multiple workers concurrently. The implementation uses the River queue library, which is a transactional Postgres-backed job queue specifically designed for Go and reached its version one stability during the project window.

The job kinds that run on the queue are enumerated as follows. Read model projection jobs consume access decision events and update the audit read model; consume member, credential, and policy events and update the corresponding read models; and run continuously while the system is active. Webhook delivery jobs are enqueued whenever an event matches a subscription filter and execute the delivery with retry logic. Tenant data export jobs are enqueued on user request and run to completion of the export. Scheduled cleanup jobs run on a recurring schedule and perform housekeeping tasks such as purging expired idempotency cache entries, expired sessions, and dead-letter queue entries past their retention window.

The singleton coordination requirement applies to scheduled cleanup jobs and to certain projection jobs that must not run on multiple workers concurrently. The system supports any number of application instances running concurrently for horizontal scaling, but some jobs must run on exactly one instance at a time. The coordination is implemented through Postgres advisory locks, keyed on the job kind. Each scheduled cleanup job, when picked up by a worker, attempts to acquire a Postgres advisory lock named after the job kind. If acquisition succeeds, the worker proceeds with the job and releases the lock on completion. If acquisition fails, the worker concludes that another instance is already running this job and exits without doing the work.

The retry handling for jobs is provided by River itself, which supports configurable retry counts, exponential backoff, and dead-letter queues per job kind. The retry configuration is set per job kind based on the kind's specific characteristics. Webhook delivery jobs use the retry configuration described in the webhook capability section. Projection jobs retry indefinitely with exponential backoff because dropping a projection update would leave the read model permanently inconsistent; if a projection job persistently fails, an operator alert is raised so the issue can be investigated. Export jobs retry a few times then fail permanently, with the failure reported to the requesting user via the export status view.

**Dashboard touchpoints.** This capability has no direct dashboard touchpoints in the user-facing admin UI, but its effects are visible through several other capabilities. The audit read model freshness lag, observable through observability dashboards, depends on projection job health. The webhook delivery status, observable through the webhook capability's dashboard touchpoints, depends on webhook job health. The export status, observable through the export capability's dashboard touchpoints, depends on export job health.

**Patterns exercised.** This capability is the primary demonstration of background job processing via the River queue library, and contributes to the distributed coordination via Postgres advisory locks pattern through the singleton job coordination mechanism.

**Scenarios served.** This capability is exercised by use case scenarios one through six insofar as nearly every scenario produces events that flow through projection jobs, and scenarios one, four, and six explicitly produce work that runs as background jobs (webhook deliveries for scenarios one and four, export packaging for scenario six).

**Explicitly out of scope.** A custom job queue implementation, a job authoring DSL for tenants to define their own scheduled jobs, integration with external task schedulers such as Apache Airflow, and a job execution UI in the admin dashboard for inspecting all jobs in the system are all excluded. River as configured is the upper bound of job processing capability in this version.

## 14. Admin dashboard: cross-cutting concerns

This section covers the elements of the admin dashboard that do not belong to any individual capability area described above. Per the hybrid dashboard treatment fixed at the start of this document, each capability area carries its own Dashboard touchpoints sub-section describing the user interface surfaces that interact with that capability; this section covers the elements that surround and unify those individual surfaces.

The information architecture of the dashboard is organized as a single top-level navigation with seven entries: Home, Members, Credentials, Doors, Policies, Audit, and Settings. The Home entry displays the real-time access event feed described in the access authorization capability section, alongside summary tiles showing counts of currently online doors, currently active members, and recent decision rate. The Members, Credentials, Doors, Policies, and Audit entries each lead to the index page of the corresponding capability area. The Settings entry is the parent of several sub-pages: Tenant Settings, User Management, Webhook Subscriptions, and Data Export. The choice to nest these four under Settings rather than promoting them to top-level is made because they are operations performed less frequently than the day-to-day capabilities and because their grouping reduces the cognitive load of the primary navigation.

Authentication flow at the dashboard level proceeds as follows. An unauthenticated user accessing any dashboard URL is redirected to the login page. The login page accepts the user's email and password, submits them to the authentication endpoint, and on success receives a session cookie that the browser persists. After login, the user is routed to the dashboard home unless their original request URL was for a specific other page, in which case they are routed to the original target. Logout invalidates the session at the server and clears the cookie at the browser. Session expiration during active use is handled by detecting a 401 response from any API call, displaying a session-expired notice, and offering a re-login button that returns to the original target after re-authentication.

Permission-aware rendering is a cross-cutting concern that touches every page in the dashboard. The dashboard receives the current session's role on every page load and renders only the controls that the role is permitted to use. An auditor-role user sees read-only views of members, credentials, doors, policies, and audit logs but no creation or modification controls. A manager-role user additionally sees the controls for creating and modifying members, credentials, and policies, and configuring webhook subscriptions. An owner-role user additionally sees the controls for managing other administrators, configuring tenant settings, and initiating data exports. The permission checks are performed both client-side (to control rendering) and server-side (to enforce). The server-side check is the authoritative one; the client-side check is a usability optimization that prevents the user from clicking a control they cannot use.

Empty states, loading states, and error states are handled consistently across the dashboard, in line with the real-data-layout principle. Every list view has an explicit empty state with a sensible message and, where appropriate, a call-to-action to create the first entity of the relevant type. Every page that fetches data shows a skeleton-based loading state during fetch, with the skeleton shape matching the eventual content shape so the layout does not shift when content arrives. Every page that may receive an error response displays the error inline within the page, with a sufficient message to identify the cause and, where the cause is recoverable, a retry button. Error boundaries at the page level prevent any single page-level error from breaking the dashboard chrome.

The real-time update strategy for views that should reflect live state is intentionally simple and explicit. The real-time access event feed on the home page uses Server-Sent Events, with the client connecting to a dedicated SSE endpoint that streams new access events for the tenant as they occur. The door status indicators on the home page and door list page poll the door status endpoint every ten seconds. The audit query view and other less-real-time views do not update without explicit user refresh. The mix of SSE for the highest-attention view and polling for moderate-frequency views avoids the complexity of a single universal real-time mechanism while providing the responsiveness the user expects on the views where it matters most. The exact technical mechanism of the SSE implementation is deferred to the System Design document.

The visual design language of the dashboard follows the brand palette and the anti-AI-aesthetics principle from the author's preferences. The primary accent color of imperial purple is applied to interactive elements such as buttons, links, and focus rings. The neutral ink palette is used for body text and structural elements. Semantic colors are reserved for status states: the safe-verified sage for granted decisions and active states, the critical magenta-plum for deny decisions and warning states, and the surface tints for background shading. Typography uses Inter as the primary system font, with a strict typographic hierarchy and minimal reliance on iconography. Layout is dense and functional with tight padding and sharp borders, deliberately rejecting the bubbly mobile-app aesthetic that pervades default AI-generated interfaces. The shadcn-ui component library provides the foundational accessible primitives that are then themed to match the brand palette through the library's Tailwind configuration overrides.

**Patterns exercised.** The cross-cutting dashboard concerns themselves exercise contract testing with versioned API schemas in OpenAPI 3.1 because the dashboard consumes the API through generated TypeScript clients derived from the same OpenAPI schema that drives the server-side validators. The permission-aware rendering exercises the role-based access control pattern from the dashboard side, complementing the server-side enforcement.

## 16. Out-of-scope features at the feature level

The PRD section six identified the high-level categories of work that are out of scope: real hardware, mobile app, payment processing, GDPR certification, production deployment infrastructure beyond Docker Compose, customer-facing commercial materials, and any real customer data. This section restates and refines that list at the feature level, identifying specific features that a reader of this PFD might expect to find and explaining their absence.

A member self-service portal where the member can update their own contact details, view their own access history, or manage their own credentials is out of scope. Members do not authenticate to OpenGate in this version; they are administrative records only. A mobile app for members to unlock doors via Bluetooth Low Energy or to receive push notifications is out of scope. The included frontend is an admin dashboard only. A payment status integration where the system tracks whether a member's dues are paid, and an automatic suspension flow that triggers when payment is missed, are out of scope; the system does not know what gym members pay or whether they pay.

A photo-on-file storage system where each member has an associated face photograph for visual identification by gym staff is out of scope. While such a feature would be natural in a real product, it does not exercise any of the twelve architectural patterns beyond what is already covered, and storing photographs would introduce data-handling concerns disproportionate to its contribution to the hiring signal. A biometric access control mode where the credential is replaced by a fingerprint or face scan is out of scope for the same reason and also because GDPR Article 9 treats biometric data as a special category requiring elevated protections that a portfolio sample is not equipped to demonstrate authentically.

A multi-language internationalization layer in the dashboard is out of scope. The dashboard is rendered in English only. A reader who is unable to use the dashboard in English is outside the target audience for the portfolio artifact. A scheduled report subscription system where compliance reports are emailed weekly is out of scope; reports are produced on demand from the audit query view only. A natural-language query interface for compliance questions, where the user asks a question in English and the system translates it to an audit query, is out of scope.

A multi-tenant administrator switching feature, where a single human identity can have administrator accounts at multiple tenants and switch between them within a single browser session, is out of scope. Each session belongs to exactly one tenant, and switching requires a separate logout and login. A federated identity layer integrating OpenGate with external identity providers such as Google Workspace, Microsoft Entra ID, or Okta is out of scope. Authentication is local username-and-password only.

An anti-passback rule system where a credential cannot be used twice on the same door without an intervening exit, an occupancy limit system where access is denied to over-full areas, and a two-person-rule system where access requires the co-presence of two members are all out of scope. These are real features in commercial access control products but each adds significant decision-logic complexity without exercising any architectural pattern not already covered by the simpler decision algorithm.

A real-time alerting layer where operators are paged on specific event types or specific anomalies in the access stream is out of scope. The observability stack supports such alerting as a future extension via Prometheus alert rules and an external alert manager, but no such rules ship with the project. An integration with email or SMS providers for outbound communication to members, administrators, or external parties is out of scope. The system communicates with external systems only through webhooks.

A re-import tool for tenant data exports, while the export format is documented as re-importable, is out of scope as a shipping feature. The format documentation is sufficient for a future engineer to write a re-import tool, but no such tool is included.

## 17. Deferred decisions

This document, like the PRD before it, deliberately defers certain decisions to the documents below it in the documentation set. Decisions deferred from this PFD to subsequent documents are listed here for visibility.

The exact mechanism by which the real-time access event feed on the dashboard home page is implemented at the transport layer, beyond the high-level decision that Server-Sent Events are used, is deferred to the System Design document. The choice between connecting the SSE endpoint directly to a Postgres LISTEN/NOTIFY channel, polling the event store on a short interval, or maintaining an in-process pub-sub layer affects the document and the implementation.

The exact projection strategy for each read model, including whether each read model is materialized synchronously within the command transaction or asynchronously by a projector job, is deferred to the System Design document. The general split has been described: read models that must be immediately fresh for subsequent reads in the same command flow are synchronous, while read models that serve broader queries are asynchronous. The per-read-model choice is left to the next document.

The exact data shapes of each read model table, including the indexes that support each query pattern, are deferred to the Database Schema document. This PFD describes which queries each read model supports but does not specify the column definitions or index choices.

The exact information architecture of the dashboard at the level of specific page layouts, form field arrangements, and button placements is deferred to a low-fidelity wireframe pass that will accompany the System Architecture document. This PFD specifies the navigation entries and the high-level content of each page but not the visual layout.

The exact set of OpenTelemetry semantic conventions used by the spans the system emits, including the span names and attribute keys, is deferred to the System Design document.

The choice of Argon2id parameters for password hashing, including memory cost, time cost, and parallelism, is deferred to the System Design document.

The exact algorithm for credential identifier generation in the backend-generated mode is deferred to the System Design document. The advisory lock coordination has been specified; the identifier-generation strategy itself remains open.
